# 玩家管理工具：黑名单、禁言、封号

## 玩家管理工具概述

本节将带你了解Warcraft III地图中**玩家管理工具**的基本概念。学完本节后，你将理解为什么需要管理玩家，以及如何在你的地图中实现基本的玩家控制功能。

### 为什么需要玩家管理工具

当你制作一张多人RPG地图时，地图会在**战网**或**局域网**上供其他玩家加入。随着玩家数量增加，难免会遇到一些特殊情况：恶意破坏游戏环境的玩家、刷屏干扰其他人的玩家、或者严重违规需要被移除出游戏的玩家。[^1]

这时，你就需要**玩家管理工具**来帮助维持游戏秩序。简单来说，玩家管理工具就是一套用来控制玩家行为的系统，包括：
- **黑名单**：永久禁止某些玩家进入你的地图
- **禁言**：限制违规玩家在游戏中发送消息
- **封号**：在当前游戏回合中强制移除问题玩家

> **💡 新手提示**：作为地图作者，你不需要实时盯着每个玩家。通过预设好的管理规则，系统可以自动处理大部分常见问题。

### 常见问题与解决方案

| 常见问题 | 解决方案 |
|---------|---------|
| 玩家恶意掉线/退出 | 设置超时检测 + 自动踢出 |
| 玩家刷屏或发送不当言论 | 启用禁言系统 |
| 玩家使用外挂或作弊 | 黑名单记录 + 封号处理 |

在Warcraft III World Editor中，这些功能通常通过**触发器**（Trigger）系统来实现。触发器你可以理解为"当满足某个条件时，自动执行的一系列动作"。[^1]

### 小结

本节我们认识了玩家管理工具的三个核心功能：黑名单、禁言、封号。在接下来的章节中，我们将学习如何在World Editor中具体配置这些功能，让你的地图拥有完善的玩家管理体系。

## 黑名单系统（Blacklist）

本节将教你什么是黑名单系统，以及如何在地图中实现禁止特定玩家进入的功能。学完本节后，你将能够保护你的RPG服务器不被恶意玩家捣乱。

### 操作步骤

1. **第一步：理解黑名单的核心原理** — 黑名单本质上是一个"玩家名字列表"（变量的一种，叫做"字符串队列"）[^1]。当玩家尝试进入游戏时，触发器会检查他的名字是否在这个列表里。如果在，就拒绝他进入。

2. **第二步：创建黑名单变量** — 在触发器编辑器中，点击"变量"按钮（左侧列表），新建一个变量：
   - 变量类型选择：*字符串*（处理玩家名字）
   - 变量名称建议写成：`Blacklist_Players`[^1]

3. **第三步：编写检测触发器** — 新建一个触发器，命名为"黑名单检测"：
   - 事件：*玩家 - 玩家进入游戏*
   - 条件：*玩家是真的人类玩家吗？*
   - 条件：*玩家名字在 Blacklist_Players 列表中吗？*（使用"列表 - 列表包含"条件）
   - 动作：*游戏 - 显示对话框给玩家"你已被禁止进入此服务器"* → *游戏 - 让玩家离开游戏*[^1]

4. **第四步：手动添加玩家到黑名单** — 另一种方式是创建一个管理触发器，管理员输入特定指令（如`-ban 玩家名`）时，自动将该玩家名字加入列表。

> **💡 新手提示**：字符串必须完全匹配，包括空格和大小写。建议在复制玩家名字时使用`Ctrl+C`精确复制，避免手动输入错误。

> **⚠️ 常见错误**：新手常忘记设置"玩家离开游戏"的动作，导致被封禁的玩家只是看到错误提示，但仍然可以继续游戏。请务必在动作中添加让玩家强制退出的代码。

### 小结

完成以上步骤后，你的地图就具备了黑名单功能。当你想封禁某个玩家时，只需要把他的游戏ID添加到`Blacklist_Players`变量中，下次他进入游戏时会被自动拒绝。黑名单系统是维护服务器秩序的基本工具，建议每个RPG地图都预留这个功能。

## 禁言系统（Mute）

在多人RPG游戏中，管理玩家行为是非常重要的。本节将教你如何制作一个**禁言系统（Mute）**，让管理员可以禁止某些玩家在游戏中发送聊天信息。学完本节后，你将能够创建一个可以临时或永久禁言玩家的功能。

### 禁言功能的基本设置

在制作禁言系统之前，我们需要先了解它的工作原理。禁言系统的核心思路是：**当玩家被禁言后，拦截他的聊天信息，让他发送的消息不显示给其他玩家**。

这个功能主要依靠**触发器（Trigger）**来实现。触发器就像游戏的"自动管家"，它会持续监测玩家的行为，当满足某个条件时（比如被禁言的玩家发送聊天），自动执行相应的操作（比如屏蔽这条消息）。

### 禁言命令的制作方法

#### 第一步：创建禁言变量

在制作禁言系统之前，我们需要一个**变量（Variable）**来记录哪些玩家被禁言了。变量就像一个"记事本"，可以存储玩家的禁言状态。

1. 打开触发器编辑器（Trigger Editor），点击左侧的"变量"（Variables）选项[^1]
2. 点击绿色的"+"按钮创建新变量
3. 将变量命名为 `MutedPlayers`
4. 将变量类型设置为 `玩家组（Player Group）` — 这是一个可以存放多个玩家的容器

> **💡 新手提示**：变量名建议使用英文或拼音开头，避免使用中文变量名，这在某些情况下可能导致触发器运行错误。

#### 第二步：创建禁言触发器

现在我们来制作核心的禁言触发器，它会监听聊天信息并决定是否屏蔽：

1. 在触发器编辑器中，点击左侧的"触发器"（Triggers）选项
2. 右键点击空白区域，选择"新建触发器"（New Trigger）
3. 将触发器命名为 `禁言系统`
4. 点击触发器左侧的"+"，添加新事件（Event）[^1]
5. 选择 **玩家 - 玩家输入聊天信息** 事件
6. 设置事件条件为 `聊天信息匹配 "/禁言"` 或类似的指令

#### 第三步：添加禁言判断逻辑

接下来，我们需要添加条件来判断被禁言的玩家：

1. 在触发器中添加条件（Condition）
2. 选择 **玩家组 - 玩家在玩家组中**
3. 设置条件为 `触发玩家 在 MutedPlayers 中`
4. 如果条件成立，执行 **游戏 - 显示聊天信息给所有玩家（禁用）** — 这样消息就不会显示出来

> **⚠️ 常见错误**：新手经常忘记给被禁言玩家的消息添加"不显示"的设置，导致禁言无效。一定要记得在触发器动作中选择"禁用聊天信息显示"选项。

#### 第四步：创建管理命令触发器

为了让管理员能够执行禁言操作，我们需要制作管理命令：

1. 再创建一个新触发器，命名为 `管理员命令`
2. 添加聊天事件，匹配命令前缀（如 "/admin"）
3. 添加子字符串判断，分辨是"禁言"还是"解除禁言"
4. **禁言动作**：使用 `玩家组 - 将玩家添加到玩家组` 把目标玩家加入 `MutedPlayers`
5. **解除禁言动作**：使用 `玩家组 - 从玩家组移除玩家`

### 临时禁言 vs 永久禁言

**临时禁言**和**永久禁言**的区别在于禁言状态的存储方式：

| 类型 | 实现方式 | 适用场景 |
|------|----------|----------|
| 临时禁言 | 配合**计时器（Timer）**，一段时间后自动从玩家组移除 | 小惩罚、警告玩家 |
| 永久禁言 | 玩家被加入 `MutedPlayers` 后不会自动移除 | 严重违规、作弊玩家 |

**临时禁言的实现方法**：在添加玩家到禁言列表后，同时启动一个计时器，计时结束后自动执行"从玩家组移除玩家"的操作。

### 小结

完成以上步骤后，你的地图将拥有一个完整的禁言系统：

- ✅ 管理员可以通过输入命令（如 `/禁言 @玩家名`）来禁言任意玩家
- ✅ 被禁言玩家的聊天消息不会显示给其他人
- ✅ 可以使用 `/解除禁言` 命令恢复玩家发言权限
- ✅ 可以设置临时禁言（自动解封）或永久禁言

> **💡 新手提示**：建议在游戏开始时给房主（Host）设置一个管理员权限，只有管理员才能使用禁言命令，防止普通玩家滥用禁言功能。

## 封号系统（Ban）

本节将教你制作封号功能，让管理员可以永久禁止违规玩家进入游戏。与简单的踢人不同，封号会将玩家加入黑名单，即使退出重进也无法再加入。学完本节后，你将拥有一个完整的封号命令系统。

### 封号与踢人的区别

很多新手会混淆"踢人"和"封号"这两个概念，它们有本质区别[^1]：

**踢人（Kick）**：只是把当前玩家移出游戏，类似"请离开房间"。玩家可以立即重新进入，惩罚效果几乎为零。

**封号（Ban）**：将玩家信息加入黑名单并保存。下次该玩家尝试进入时，系统会自动拒绝并显示"您已被封号"。这是真正的惩罚手段。

### 操作步骤

#### 第一步：创建封号存储结构

封号需要一个"容器"来保存被禁止的玩家信息。我们使用**数组**来存储这些数据[^1]：

1. 打开触发器编辑器（Trigger Editor）
2. 在左侧"变量"区域右键 → 新建变量
3. 变量名称填写 `BannedPlayers`（可自定义）
4. 变量类型选择 `玩家组`（Player Group）
5. 勾选"数组"选项，设置大小为 **16**（因为 Warcraft III 最多支持16名玩家）

> **💡 新手提示**："玩家组"就像一个名单，可以添加或移除玩家名称。如果不使用数组，当多个玩家同时被封号时，后面的会覆盖前面的记录。

#### 第二步：制作封号命令触发器

1. 在触发器列表中右键 → 新建触发器，命名为"管理命令-封号"
2. 设置事件：`玩家 - 玩家 1 输入的聊天内容`（匹配文字 "/ban"）
3. 设置条件：
   - `玩家1的身份` = `管理员`（或检查聊天内容是否包含"@你的ID"等验证方式）
4. 设置动作：
   ```
   玩家组 - 将 (截取的玩家) 添加到 BannedPlayers
   游戏 - 强制 (截取的玩家) 离开游戏
   显示题目：对所有玩家显示 "玩家 [名字] 已被管理员封号"
   ```

> **⚠️ 常见错误**：很多新手忘记设置"条件"检查权限，导致任何玩家都能随便封号别人。务必加上管理员身份验证！

#### 第三步：设置登录检查系统

有了封号命令，还需要一个系统来检查进入的玩家是否在黑名单中[^1]：

1. 新建触发器"登录检查"
2. 设置事件：`游戏 - 地图初始化`
3. 设置动作：
   ```
   对于每个整数 A 从 1 到 16，执行：
       如果 BannedPlayers 包含 玩家(A)，则：
           游戏 - 强制 玩家(A) 离开游戏
           显示题目：对所有玩家显示 "系统检测到封号玩家，已拒绝进入"
   ```

#### 第四步：持久化保存封号数据

上面的设置在**每次重开游戏后会失效**，封号记录全部丢失。我们需要使用**哈希表**来永久保存数据[^1]：

1. 新建变量 `BanData`，类型选择 `哈希表`（Hashtable）
2. 在"管理命令-封号"触发器中，添加动作：
   ```
   哈希表 - 保存 (玩家ID) 作为 BannedPlayers 中 (玩家号码) 的值
   ```
3. 在地图初始化触发器中，添加动作：
   ```
   哈希表 - 从文件 "banlist.txt" 读取 BanData
   ```

> **💡 新手提示**：哈希表就像一本字典，"banlist.txt"是记录本文件名。你可以自定义这个文件名，但每次保存/读取时名称必须一致。

### 小结

完成以上步骤后，你的地图将具备完整的封号管理功能：

- ✅ 管理员输入 `/ban @玩家名` 即可封号
- ✅ 被封号的玩家会被立即踢出并无法重新进入
- ✅ 封号数据保存到本地，重启游戏后依然有效
- ✅ 普通玩家无法使用封号命令

这套系统可以进一步扩展，例如添加"解封"命令（`/unban`

## 综合管理工具制作

本节将教你如何制作一套完整的**管理员工具系统**，包括菜单界面、权限控制和操作记录。学完本节后，你将能让服务器管理员方便地管理玩家、处理违规行为。

### 操作步骤

1. **第一步：创建管理员变量** — 在触发器编辑器左侧右键点击"变量"，选择"新建变量"[^1]。创建一个**布尔类型的变量**命名为"是管理员"，数组大小设为12（因为WC3最多支持12个玩家）。这个变量用来记录每个玩家是否是管理员。

2. **第二步：建立管理员识别触发器** — 新建一个触发器，命名为"检查管理员"。设置**事件**为"玩家 - 玩家进入游戏"。**条件**留空，**动作**添加：读取聊天信息、判断玩家输入的密码是否正确（可用字符串比较），如果正确则设置对应玩家的"是管理员"为真[^1]。

3. **第三步：制作管理员菜单** — 新建触发器"管理员菜单"。事件设为"聊天 - 玩家输入'!admin'"。动作包括：显示对话框给玩家，选项包含"黑名单"、"禁言"、"踢出玩家"等按钮。每个按钮对应一个子触发器[^2]。

4. **第四步：设置权限等级** — 再创建一个**整数变量**"管理员等级"。设置多个数值：0=普通玩家，1=普通管理员，2=超级管理员。在每个管理功能触发器中，用**条件**检查管理员等级是否足够，例如："整数 - 比较 (管理员等级[(玩家编号(触发玩家))]) 大于等于 (1)"[^2]。

5. **第五步：添加日志记录** — 创建"日志记录"触发器。当管理员执行任何操作时，将操作内容写入一个**字符串变量**"操作日志"，同时附上时间戳（可用游戏内部时间）和管理员名称。可以将日志保存到文本文件或显示在特定区域[^1]。

> **💡 新手提示**：建议把所有管理员相关的触发器单独放在一个文件夹里，右键触发器列表选择"新建文件夹"，命名为"管理员系统"，方便后续管理。

> **⚠️ 常见错误**：新手容易忘记为每个玩家单独设置变量值。比如设置"是管理员"时，应该用"(玩家编号(触发玩家)) + 1"作为数组索引，而不是直接写0，否则所有玩家的权限会混在一起。

### 小结

完成以上步骤后，你应该已经搭建起一个**基础的管理员系统**。玩家输入密码后可以成为管理员，打开菜单后可以对其他玩家执行禁言、踢人等操作，所有操作都会被记录下来。建议先测试基本功能是否正常，再逐步添加封号、广播公告等更复杂的权限功能。

## 最佳实践与常见错误

在本节中，你将学习玩家管理功能中最容易出错的地方，以及让系统运行更流畅的技巧。学完以后，你的地图将能更安全、稳定地管理玩家行为。

### 新手常犯的错误

1. **忘记给被封禁玩家设置"永久标记"** — 很多新手只用了踢人功能，但没有用变量（用于存储数据的"容器"）记住被封禁的玩家ID[^1]。结果玩家重连后又可以进来捣乱。

2. **禁言触发器条件写得太宽泛** — 比如把"所有玩家"都禁言了，导致管理员自己也发不了消息。

3. **没有备份黑名单数据** — 辛辛苦苦建立的玩家记录，因为地图重载或版本更新全部丢失。

> **⚠️ 常见错误**：新手经常把"触发器"的"条件"和"动作"搞混。条件是指"什么时候执行"，动作是指"具体做什么"。写反了会导致功能完全失效。

### 优化建议与性能考虑

1. **使用整数变量而非字符串来存储玩家ID** — 整数比较比字符串比较快得多[^1]，能让你的管理功能响应更迅速。

2. **把常用玩家数据存到哈希表中** — 这样即使地图重新加载，数据也不会消失。

3. **避免使用"每隔X秒"这种无限循环检测** — 改为用事件驱动（比如玩家聊天时），能大幅降低卡顿。

> **💡 新手提示**：在触发器中，点击绿色的"事件"、黄色的"条件"、红色的"动作"可以分别添加不同类型的代码块，顺序不能乱。

### 社区经验分享

根据许多地图作者的实战经验[^2][^3]：

- **分级管理很重要** — 把玩家分为"普通玩家""VIP""管理员"三个等级，用不同的触发器处理不同权限。

- **保留申诉渠道** — 即使封禁了玩家，也建议在地图说明里留下联系方武，方便误封时解封。

- **日志记录不可少** — 用"聊天记录"触发器把违规玩家的行为记到文件中，出了问题有据可查。

### 小结

现在你已经掌握了玩家管理功能的核心要点。记住：先小范围测试再全服开放，数据要记得备份，条件逻辑别写反。完成以上注意事项后，你的地图就能更好地维护秩序，让正常玩家有更好的游戏体验。

# 多人游戏同步机制

## 什么是多人游戏同步

在本节中，你将了解什么是多人游戏同步，为什么它在多人对战中至关重要，以及本地执行与网络执行的区别。学完本节后，你将能够理解为什么你的地图在单人测试时完美运行，但在多人联机时会出现各种奇怪的bug。

### 同步的定义与重要性

**同步（Synchronization）** 是指在多人游戏中，确保所有玩家看到相同游戏状态的过程。当你和朋友一起玩地图时，你们需要看到完全一样的单位位置、血量、技能效果——这就是同步要解决的问题。

在魔兽争霸III中，当一张地图支持多人游戏时，所有玩家的电脑（包括主机房主）会通过网络连接在一起。**触发器（Trigger）** 是你在World Editor中创建的自动化逻辑，比如"当单位死亡时给予击杀者金币"。问题是：如果每个玩家的电脑都独立运行这段代码，可能会因为网络延迟、计算顺序等原因，导致每个玩家看到的结果不一致[^1]。

这就是为什么魔兽争霸III采用了"主机权威"模式：**主机（Host）** 的电脑负责运行大部分关键逻辑，然后通过网络将结果发送给其他玩家。同步机制就是确保这些结果能够正确、高效地传递给每个客户端。

> **💡 新手提示**：可以把多人游戏想象成一个微信群聊——主机是群主，负责整理大家的信息并广播给所有人；其他玩家是群成员，收到群主的消息后同步更新自己的"聊天记录"。

### 本地执行与网络执行的区别

在World Editor中创建触发器时，你需要理解两种执行模式的区别：

**本地执行（Local Execution）**：代码只在**当前玩家的电脑**上运行，不会同步给其他玩家。比如显示一个只有你自己能看到的对话框，或播放只有你自己能听到的音效。这适合做个人UI反馈。

**网络执行（Network Execution）**：代码会通过网络发送给所有玩家，所有人同时执行。比如修改一个单位的属性、创建/删除单位、改变游戏状态等。只有设置为"网络同步"的触发器动作才会被所有玩家同步执行[^1]。

**关键区别**：
- 本地执行：快，无需等待，但其他玩家看不到效果
- 网络执行：需要网络传输，有延迟，但保证所有玩家看到相同结果

在设计多人地图时，你需要在两者之间做出选择：某个功能是否需要所有玩家都能看到？如果是，就使用网络执行；如果只是给单个玩家反馈，就用本地执行。

> **⚠️ 常见错误**：新手经常忘记给触发器添加网络同步设置，结果导致只有主机能看到效果，客户端玩家却看不到任何变化。检查你的触发器动作属性，确保"网络启用"选项被勾选。

### 小结

现在你明白了：多人游戏同步就是确保所有玩家看到相同游戏状态的技术。在魔兽争霸III中，主机负责运行关键逻辑并广播结果，这就是"主机权威"模式。本地执行只在当前玩家电脑上运行，而网络执行会同步给所有玩家。理解这个区别，是你制作多人地图的第一步——只有正确使用网络执行，你的多人地图才能正常工作。

## Warcraft III 的同步架构

本节将帮助你理解 Warcraft III 多人游戏背后的**同步原理**，以及主机（Host）和客户端（Client）之间如何分工协作。学完本节后，你将能够判断你的触发器代码在多人模式下是否会出现"不同步"问题，并知道基本的修复思路。

### 主机与客户端的角色分工

在 Warcraft III 的多人游戏中，**一台电脑充当"主机"（Host），其他电脑都是"客户端"（Client）**[^1]。这就像一个课堂——主机是老师，客户端是学生。老师说什么，同学们就跟着做什么。

**具体角色区别如下：**

- **主机（Host）**：游戏的"权威中心"，负责处理游戏逻辑、计算伤害、判断胜负等核心运算。所有玩家的指令最终都由主机汇总后统一执行。
- **客户端（Client）**：负责显示游戏画面、播放音效、响应玩家的本地操作（如移动镜头、点击单位）。客户端本身不进行游戏逻辑判断，只是把玩家的操作发送给主机。

> **💡 新手提示**：在局域网或战网联机时，创建房间的玩家自动成为主机。如果主机退出游戏，整个房间会解散。

### 触发器在多人模式下的执行流程

当你在 World Editor 中编写触发器时，**它会在每一台玩家的电脑上分别执行**。这听起来很方便，但同时也带来了最大的挑战——**同步问题**。

**同步触发器的执行流程如下：**

1. **主机执行**：触发器首先在主机上运行，计算出结果（比如某个单位获得100点生命值）。
2. **结果广播**：主机将触发器的"执行结果"发送给所有客户端。
3. **客户端同步**：客户端收到结果后，"同步"显示相应的变化。

问题是：**如果触发器里有"随机数"、"读取本地玩家输入"或"依赖游戏时间"的逻辑，不同电脑可能计算出不同结果**——这就是"**不同步（Desync）**"，会导致游戏崩溃或各玩家看到的情况不一致。

> **⚠️ 常见错误**：新手常在触发器里使用"获取随机整数"后直接作用于游戏逻辑，这在多人模式下会导致各玩家看到不同的结果。解决方法是将随机数计算放在主机的触发里，然后用"**同步整数**"或"**同步布尔值**"动作将结果发送给所有客户端[^1]。

### 小结

理解主机与客户端的分工，以及触发器的同步执行原理，是制作多人 RPG 地图的**必备基础**。只要记住：**游戏逻辑在主机上运算，结果通过同步机制广播给所有客户端**，你就能避免大多数不同步问题。接下来你可以尝试创建一个简单的双人地图，验证你的触发器是否能在多人模式下正常运行。

## 常见的同步问题原因

在多人游戏中，"同步"意味着所有玩家看到完全相同的游戏画面和事件。如果某个玩家的游戏状态和其他人不同，就会出现"不同步"（Desync）问题，导致游戏崩溃或出现奇怪的行为。本节将帮助你识别三种最常见的同步问题原因。

### 1. GetLocalPlayer() 的错误使用

`GetLocalPlayer()` 是JASS脚本中的一个函数，用于获取"当前正在操作这台电脑的玩家"。**新手最常犯的错误是在触发器中用它来控制只有自己能看到的画面**，比如这样写：

```jass
if GetLocalPlayer() == Player(0) then
    // 只有玩家1会执行这段代码
    SetUnitPosition(someUnit, 100, 200)
endif
```

问题在于：当玩家1的单位移动到(100, 200)时，其他玩家不知道这个单位去了哪里，因为他们的客户端没有执行这段代码。这就是不同步的根源。

> **⚠️ 常见错误**：很多新手以为"只有本地玩家能看到"和"只有本地玩家能操作"是同一回事。实际上，前者会直接导致不同步，而后者是正确的做法。记住：**永远不要用 GetLocalPlayer() 来改变游戏中的单位、物品或技能状态**。

### 2. 随机数与计时器的不同步

War3的"随机数"（Random Number，简称RN）系统并不是真正的随机。它使用的是"伪随机数生成器"（Pseudo-Random Number Generator），每次随机都会消耗一个"种子"，而这个种子在每个玩家的客户端上是**独立推进**的[^1]。

想象你和朋友同时玩游戏，你用"闪电链"攻击了一个单位（触发了一次随机伤害），你的客户端计算种子+1，但朋友的客户端种子还是原来的值。几个回合下来，你们的随机数序列就完全错位了。

计时器也有类似问题：如果你的触发器依赖"每0.01秒检查一次"，不同玩家的电脑性能不同，触发时机也会有细微差异。

> **💡 新手提示**：尽量使用War3自带的"等待"动作而不是自定义计时器。自带等待会根据游戏帧率同步，不容易出问题。

### 3. 浮点数计算导致的数据差异

浮点数（就是带小数点的数字，比如 3.14159）在不同电脑上进行计算时，**极微小的精度差异**可能导致最终结果不同。例如：

```jass
local real a = 10.0 / 3.0
local real b = a * 3.0
// 在不同电脑上，b 可能等于 9.9999999 或 10.0000001
```

这种微小差异累积起来，就会让某个玩家的单位位置、伤害数值和其他人产生偏差，最终导致不同步。

### 小结

你现在知道了三种最常见的同步问题：
- ❌ 用 `GetLocalPlayer()` 改变游戏状态
- ❌ 在多人游戏中随意使用随机数
- ❌ 依赖精确浮点数计算

**下一步**：请查看"如何避免同步问题"章节，学习正确的多人游戏制作方法。

## 正确的同步实践方法

在多人游戏中，所有玩家的电脑需要保持"步调一致"。本节将教你如何使用WC3提供的同步功能，确保每个玩家看到相同的游戏状态。学会后，你的地图将不再出现"你看到的是A，他看到的是B"这种尴尬情况。

### 可靠动作与不可靠动作的使用

在WE中，触发器动作分为"同步"和"非同步"两类。**同步动作**（可靠动作）会强制所有玩家的客户端执行相同代码；**非同步动作**（不可靠动作）则只在本地执行，不同玩家的结果可能不同[^1]。

#### 操作步骤

1. **打开触发器编辑器** — 在WE顶部菜单点击"触发器"图标（或按F4）
2. **找到"游戏-同步"分类** — 在左侧动作列表中展开"游戏"类别，你会看到"同步整数"、"同步真实数"等动作[^1]
3. **将变量包装在同步动作中** — 例如，不要直接设置变量，而是用"同步整数"动作包裹你的赋值操作
4. **理解哪些是本地动作** — "创建单位"、"移动单位"等动作默认是本地的，只有被同步动作触发时才会统一执行

> **⚠️ 常见错误**：新手以为设置了变量就能同步，其实"设置变量"这个动作不会自动同步。变量同步必须依赖"同步整数/真实数/字符串"等专门动作[^1]。

### 同步触发器与本地触发器的选择

#### 操作步骤

1. **判断触发器的用途** — 问自己："这个逻辑是否影响所有玩家？"如果答案是"是"，就用**同步触发器**
2. **创建同步触发器** — 在触发器名称上右键 → "新触发器"，然后在"事件"中使用"玩家组中的玩家进行游戏"或"所有玩家进行游戏"作为事件源
3. **本地触发器的使用场景** — 只影响单个玩家的UI显示、视觉效果等，用本地触发器即可，性能更好
4. **混合使用** — 一个常见模式是：用本地触发器检测玩家输入，然后用"调用同步触发器"来统一处理结果

> **💡 新手提示**：简单记忆——"多人互动用同步，单人独享用本地"。比如伤害计算、物品分配必须同步；播放音效、特效显示可以本地。

### 玩家组的正确使用方式

**玩家组**是同时操作多个玩家的工具，但如果使用不当，会导致同步问题。

#### 操作步骤

1. **创建玩家组** — 在触发器中："设置 -> 玩家组"，选择需要的玩家（如"所有玩家"或特定玩家）
2. **对玩家组执行动作时使用循环** — 不要期望一个动作自动处理所有玩家，需要用"对于每个整数A从1到(玩家组中的玩家数量)"循环
3. **动态修改玩家组要谨慎** — 如果游戏中途加入/离开玩家，用"同步整数"确保所有客户端获得相同的玩家组状态
4. **避免在循环中创建/销毁玩家组** — 在触发器开头创建一个玩家组变量，结尾统一销毁，不要反复创建

> **⚠️ 常见错误**：新手直接在玩家组上执行动作，忽略了不同客户端的玩家组可能状态不同。正确做法是：先同步玩家组的变动，再用循环逐个处理。

### 小结

完成以上步骤后，你应该：
- ✅ 知道在需要同步的地方使用"同步整数/真实数"等动作
- ✅ 能区分哪些触发器需要同步，哪些可以本地执行
- ✅ 掌握用循环配合玩家组进行多玩家操作
- ✅ 避免"各玩各的"导致的游戏崩溃或作弊

记住：**多人游戏的同步原则是"宁可多同步，不可漏同步"**。当你不确定时，保守地多使用同步动作，比出问题后再调试要省心得多。

## 同步问题的调试与检测

本节将学习如何在Warcraft 3编辑器中识别和调试多人游戏中的同步问题。学完本节后，你将能够独立测试你的地图，并发现潜在的同步错误。

### 同步错误的识别方法

1. **观察玩家行为差异** — 在本地测试（按住"Test Map"）时按住F9打开debug窗口，观察不同玩家控制单位的移动是否一致[^1]

2. **使用"同步随机数"调试** — 在触发器中添加显示随机数结果的聊天信息，确保所有玩家收到相同的随机数序列

3. **检查单位位置同步** — 让所有玩家同时放置单位，通过聊天显示各单位坐标，对比数值是否完全一致

### 测试与验证技巧

1. **启用记录模式测试** — 在编辑器菜单中选择"Scenario" → "Test Map"，使用2个以上的"Computer"玩家加入游戏[^3]

2. **分步打印调试法** — 在复杂触发器中插入"Display Text Message"（显示文本消息）到屏幕，标记每个关键步骤的执行情况[^1]

3. **局域网联机实测** — 本地测试无法完全模拟真实网络延迟，必须用两台电脑通过局域网或战网实际联机测试[^2]

> **⚠️ 常见错误**：新手只在本地图编辑器里反复按F10测试就觉得万事大吉。实际上本地测试是单线程模拟，永远不会触发真正的网络同步问题。**必须找另一台电脑实际联机**，才能发现真实的同步bug。

> **💡 新手提示**：如果发现某个触发器在联机时表现异常，尝试把它拆分成多个小触发器分别测试，这样能快速定位问题出在哪个环节。

### 小结

完成以上步骤后，你应该掌握了基本的同步调试方法：先通过观察玩家行为差异发现问题，再用分步打印法缩小范围，最后通过真实联机测试确认修复效果。建议每次修改同步相关触发器后都进行联机测试，养成良好习惯。

## 高级同步技术

本节将深入讲解Warcraft III多人游戏中的**确定性同步原理**和**自定义同步实现方法**。学完本节后，你将理解为什么多人游戏会出现"不同步"问题，并学会使用触发器同步来保证游戏数据一致。

### 确定性系统的设计原则

在多人游戏中，所有玩家的电脑必须计算出**完全相同的结果**，否则就会出现不同步（Desync）[^1]。想象你和朋友同时玩，但如果你们的"世界"不一样，游戏就会崩溃。

1. **使用整数而非实数** — 避免使用`real`类型进行关键计算，因为小数运算在不同电脑可能产生微小差异[^2]
2. **避免依赖执行顺序** — 循环和条件判断的顺序必须对所有玩家一致，不能用"先到先得"的逻辑[^2]
3. **使用同步触发器** — 通过`Trigger Execute`或`PolledWait`等同步函数确保关键代码同时执行[^1]

> **⚠️ 常见错误**：新手常使用`GetRandomInt`在所有客户端生成随机数，这会导致每个玩家的"随机"结果不同。解决方法：先由主机生成随机数，再通过触发器同步给所有玩家。

### 自定义同步的实现方法

当你需要同步自定义变量或复杂数据时，需要手动传递数据：

1. **创建同步触发器** — 新建一个"共享同步"触发器，设置为"服务器/客户端"模式[^1]
2. **使用游戏缓存传输数据** — 利用`Save...Handle`将数据存入缓存，再用`Load...Handle`读取[^2]
3. **广播关键变量** — 在主机修改变量后，通过`BJDebugMsg`或自定义函数通知所有客户端

> **💡 新手提示**：最简单的方法是使用"共享单位"或"共享单位组"功能，让WE自动帮你处理同步。但如果你需要同步自定义的整数或字符串，必须手动写触发器。

### 小结

完成以上步骤后，你应该理解：
- 确定性是多人游戏同步的基石，必须确保所有计算对每位玩家完全一致
- 避免使用浮点数进行关键逻辑，优先选择整数运算
- 使用WE提供的同步功能或自定义触发器来传递关键数据

如果遇到持续不同步，优先检查是否有未同步的随机数、未对齐的循环、或依赖本地执行顺序的逻辑。
